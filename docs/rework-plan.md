## Принцип
Скилл = итеративный дебат Claude+Codex над **одним артефактом**, который улучшается по раундам. Артефакт **всегда спека** (ТЗ / план / дизайн-документ **любой** предметной области — разработка, архитектура, медицина, производство и т.д.). Три варианта входа различаются тем, как спека засевается; **петля раунда у всех одна**. Скилл **на каждом раунде редактирует именно спеку (артефакт)** — это и есть его работа. **Внешний предмет**, который спека описывает (код, существующая система и т.п.) во время дебата НЕ трогаем; для кода спека модификаций применяется к коду уже ПОСЛЕ дебата (если есть доступ). Когда предмет работы — сама спека/ТЗ (вариант 3), она же и есть редактируемый артефакт. Существующий **prompt-only** режим (краткий совет без итерируемого артефакта) сохраняется как есть.

## Три варианта входа (различаются посевом спеки; В2 — ещё доп. обязательствами)
- **Вариант 1 — TASK → SPEC.** Вход: задача/вопрос. Claude пишет первоначальную спеку решения; может ссылаться на код/материалы проекта, если релевантно (напр. проектируем новый сервис, изучив код существующих — рефакторинг не нужен, но код изучить надо).
- **Вариант 2 — CODE → CHANGE-SPEC.** Вход: код + задача (рефакторинг/оптимизация/доработка). Claude изучает код и пишет первоначальную **спеку модификаций**, над которой далее идёт работа. При этом в каждом раунде помимо анализа спеки проводится и анализ кода + требований пользователя. **Разведка кода по раундам:** раунд 1 — глубокая (что изучено / что опущено + ключевые факты фиксируются в спеке и переезжают между раундами); далее — точечная, т.к. код во время дебата не меняется (детали — в «Передача предметного материала Codex'у»). Опциональная **пост-дебатная** имплементация.
- **Вариант 3 — SPEC → SPEC.** Вход: готовая спека + задача. Артефакт = эта спека.
- Во всех: контекст, путешествующий с раундами = **постановка задачи + спека + предметный материал (код / иные источники, напр. документы), если есть и релевантен + файл дебатов/решений (state-файл: settled-вердикты прошлых раундов)**.
- **[Codex #3]** Точная формулировка: *петля предложения→слияние→persist одна для всех; варианты различаются главным образом посевом спеки; В2 дополнительно требует опциональную пост-дебатную имплементацию.*

## Выбор режима: prompt-only vs итерируемая спека (В1–В3)
Решает **Claude после первоначальной оценки объёма** — широты и сложности задачи, не по ключевым словам. **Оценка объёма = поверхностный осмотр** (структура, размер, число файлов/компонентов, есть ли нетривиальный дизайн), **не глубокое чтение**: глубокая разведка происходит уже ВНУТРИ выбранного режима (см. «Передача предметного материала Codex'у») — это исключает двойную работу. Затем выбор:
- **prompt-only** (краткий совет, один пасс, без файла и раундов) — когда результат это совет/сравнение, который пользователь применит сам; артефакт итерировать не нужно; осмотр показывает, что вопрос закрывается за один проход. Пример: «вот вопрос — посоветуйся с codex».
- **Итерируемая спека (В1/В2/В3)** — когда осмотр вскрывает широту/сложность: надо изучить несколько файлов/компонентов или существенный материал, дизайн с рядом связанных решений, либо пользователь хочет итерацию / письменную спеку. Пример: «посмотри, оптимизируй» по большому объёму кода в нескольких файлах — лёгким режимом не обойтись: засеваем спеку и идём раунды.
- Выбранный режим **назвать одной строкой**. Если уже в prompt-only вскрылось, что объём больше ожидаемого — **закончить текущий пасс**, затем предложить перейти к итерируемой спеке (не обрывать на полуслове).

## Минимальная форма спеки — [Codex #2; quality-gate]
Чтобы дебат не сходился на недоспецифицированном, посеянная спека обязана концептуально содержать:
- **Общая (все варианты):** goal · non-goals · assumptions · constraints · acceptance criteria · open questions · какой материал изучен и что намеренно опущено (если материал изучался).
- **В2 дополнительно:** предлагаемые изменения по файлам/компонентам · что в поведении сохраняется vs меняется · риски/миграции · acceptance-проверки/тесты.
Крошечные задачи — сжатая версия, но поля концептуально присутствуют. Если **материальные** факты отсутствуют и влияют на спеку — **спросить пользователя до дебата**, не дебатить выдуманное.
**Применимость по вариантам:** форму обеспечивает Claude, когда **сам пишет** спеку (В1/В2). Для В3 (готовая спека) это **не предусловие** — отсутствующие поля становятся находками дебата, а не блокером, и Claude не переписывает вход до обсуждения. **Но если вход фундаментально пуст** (заглушка / одна строка, а не реальная спека) — это уже не В3, а по сути В1: Claude засевает спеку сам (запросив недостающие материальные факты у пользователя), а не дебатит пустой артефакт.

## Единый цикл раунда
1. **Артефакт на начало раунда** (В1: черновик Claude; В2: спека модификаций Claude; В3: заданная (р.1) / обновлённая (далее) спека).
2. **Независимые предложения обеих моделей.** Claude и Codex независимо формируют каждый свой список предложений по улучшению спеки. **Независимость означает только одно: Codex не видит список предложений Claude текущего раунда** (чтобы не якориться). Общий контекст Codex получает — постановку задачи + саму спеку (в В1/В2 она и так воплощает прошлые решения Claude) + предметный контекст + settled-вердикты прошлых раундов: это артефакт и зафиксированные решения, **а не предложения текущего раунда**. Во всех вариантах Codex просим указывать не только правки к написанному, но и то, что спека упускает относительно задачи и доступного материала (альтернативы, пропущенные риски, непокрытые требования). Для В2 это особенно критично из-за анкеринга (спека = план Claude над кодом, см. блок про анкеринг).
3. **(Опция «thorough») Кросс-критика.** Каждая модель получает список оппонента и пишет ревью по каждому пункту: согласен / частично / отклонить + аргумент. Это +1 Codex-вызов (он headless, вызовы сериализуются) — включаем для сложных/спорных спек, иначе пропускаем. *(Финальная реализация: один доп. вызов по списку Claude + self-enable при own major+ — см. шапку.)*
4. **Слияние — Claude (мержер), с правом вето.** Claude сводит оба списка (+ кросс-ревью, если был) в **ОДИН консолидированный список правок**. **Оба списка — и Codex, и собственные предложения Claude из шага 2 — проходят ОДНУ процедуру:** по каждому пункту явное решение с аргументом (принял / частично / отклонил — почему) через линзу `impact × likelihood × altitude` (не глуша правдоподобные edge-кейсы); **отклонённые собственные предложения тоже попадают в отчёт.** Мёржер всегда Claude, поэтому анти-bias — симметричный аудит обоих списков (своих и Codex) + обязательная аргументация + аудит в отчёте (+ кросс-критика в thorough).
5. **Применить к спеке**; перечитать обновлённые места на самосогласованность.
6. **Отчёт:** принято (K) / частично (P) / отклонено (R), однострочные причины, прогресс по раундам, источник находки.
7. **Persist** (state-файл рядом со спекой). **Критерий сходимости:** если в раунде не принято ни одной **материальной** правки (только косметика/формулировки) ИЛИ остались только минорные предложения — явно **рекомендовать СТОП**. **[Codex #6]** По дисциплине текущего скилла: один вызов = один раунд; в отчёте **сообщить**, остались ли значимые пробелы (+ рекомендацию стоп/ещё по критерию выше) и что пользователь может запустить ещё раунд — **не продолжать без явной просьбы** (не интерактивный промпт каждый раунд). *(Финальная реализация: явная просьба нескольких раундов выполняется в одном вызове — см. шапку.)*

## Граница «дебат → имплементация» (В2) — [Codex #1]
Применение кода — **пост-дебатный execution-шаг, не часть петли дебата**. «Спека сошлась» ≠ «код работает». Поэтому:
- финальная спека модификаций включает **acceptance-проверки/тесты**, где возможно;
- если у Claude есть доступ и он применяет — далее **типовая инженерная петля**: правка → тесты/проверки → **отчёт об имплементации ОТДЕЛЬНО от вердиктов дебата**.
В конце В2 предложить (если есть доступ к коду): применить согласованную спеку к коду сейчас — иначе стоп / ещё раунд (по обновлённой спеке).

## Решение для кода (В2): артефакт = спека (Option A) — подтверждено Codex
Итерируем **спеку модификаций**, код применяем в конце (НЕ итерируем сам код по раундам). Аргументы: универсальность (одна петля); удалённый/MCP-код (репо, доступное Claude только через инструмент, до которого Codex не дотянется → спека — единственный общий артефакт, Option B невозможен); сходимость (общая спека сходится, параллельные перезаписи кода расходятся); аудит/безопасность (код меняется один раз); дебат на уровне намерения дешевле итерировать. **Option B** (итерировать сам код локально) — явный НЕ-дефолт, лишь как быстрый локальный путь для мелких конкретных правок, когда нужна имплементация, а не дебат.

## Анкеринг в В2 — (a) дефолт + триггер эскалации — [Codex #4]
В В2 спека = предложения Claude, поэтому Codex частично заякорен. **Дефолт (a):** принять — адверсариальное ревью плана ценно; Codex просим искать дыры/упущенные альтернативы/риски, не штамповать. **Эскалация к (b)** (раунд 1 — независимый анализ кода+задачи Codex'ом без черновика Claude, затем свод в спеку) **только когда** задача high-impact / неоднозначна / security-sensitive / спека — в основном необъяснённая стратегия Claude, а не ограниченный запрос пользователя.

## Передача предметного материала Codex'у — [Codex #5: обобщено на не-код]
- **Разведку ведёт Claude:** bounded, purpose-driven. Для кода — символы/callsites/тесты/конфиги. **Для не-код доменов** «контекст» = относящийся материал, который Claude изучил: ограничения, диаграммы, политики/регуляции, факты площадки, threat-model, измерения, прежние решения — и что намеренно опущено.
- **Codex получает:** точные ссылки, когда материал на ФС, где исполняется Codex; **эмбед**, когда удалённый / из MCP / нефайловый (удалённое MCP-репо → эмбед обязателен). Open-ended разведку Codex'ом не допускаем (историческое зависание).
- **По раундам (В2 / код-контекст):** код во время дебата не меняется → у Claude разведка глубокая в раунде 1, далее лишь **точечные** доразведки (проверить утверждение Codex или участок, впервые попавший в скоуп); зафиксированные в спеке «изучено / опущено + ключевые факты» переезжают между раундами, не переоткрываясь. Codex headless/stateless → каждый раунд получает тот же **курируемый срез** материала (ссылки/эмбед) + обновлённую спеку + settled-вердикты и читает строго в его пределах; срез расширяем только при расширении скоупа.
