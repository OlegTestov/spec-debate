# spec-debate — дебаты двух моделей над планом решения задачи

Скилл для **Claude Code**. Написал план решения задачи / ТЗ / спеку / технический
дизайн — запускаешь `/spec-debate`, и его начинает критиковать **вторая модель
(OpenAI Codex)**, а Claude выступает **редактором с правом вето**: тщательно проверяет каждую претензию, применяет только то, что реально улучшает,
остальное аргументированно отклоняет.

Это не только про разработку спецификаций: можно «посоветоваться с Codex» прямо по ходу
работы — скилл прогонит один раунд и вернёт независимый разбор без сохранения состояния.
Если решишь продолжить работу с Codex — автоматически создаст файл с итерациями и перейдёт
в стандартный режим дебатов.

> Критик, которого слушаются всегда, — это просто второй автор.
> Критик, с которым спорят, даёт результат лучше, чем каждая модель поодиночке.

По умолчанию один вызов = один раунд: критика → вето → правки → отчёт → сохранённое
состояние (`.<имя>.debate-state.json` рядом с файлом). Нужно несколько раундов — вызови
скилл ещё раз (продолжит с того места, где остановились), либо попроси сразу: «прогони
3 раунда» или «гоняй, пока не останется значимых замечаний». Отклонённые находки
передаются Codex'у, чтобы он не поднимал их заново. Зашитый принцип — результат **лучше**,
а не **больше**: скилл сопротивляется разрастанию сложности.

## Требования

- **Claude Code** (CLI от Anthropic).
- **OpenAI Codex CLI** — установлен и авторизован:
  ```bash
  npm install -g @openai/codex
  codex login status    # проверка авторизации (если не залогинен — codex login)
  ```
  Нужен именно *независимый* второй движок — в этом весь смысл дебатов.
- **`bash` и `pgrep`** в PATH — их использует вспомогательный скрипт. Есть по умолчанию в **macOS и стандартных Linux**.
  **Windows: используйте WSL2** (обычное Linux-окружение) — это поддерживаемый путь. Нативный Windows не поддержан:
  под Git Bash `pgrep` обычно отсутствует, поэтому скрипт честно завершается с ошибкой (fail-closed), а без Git Bash
  bash вообще не запустит скрипт.

## Данные и приватность

spec-debate отправляет **полный текст документа** (и файлы, которые Codex прочитает в рабочей папке) в OpenAI через Codex CLI. Не запускай на материалах, которые нельзя передавать в OpenAI — секреты, клиентские/NDA-данные.

## Установка

Проще всего — если твоему Claude Code разрешены shell-команды — попросить его установить скилл:

> Установи https://github.com/OlegTestov/spec-debate в ~/.claude/skills/spec-debate

Он сам склонирует репозиторий в папку скиллов. Не сработало — склонируй вручную:

```bash
git clone https://github.com/OlegTestov/spec-debate ~/.claude/skills/spec-debate
```

Папку скилла можно положить в одно из двух мест:

- `~/.claude/skills/spec-debate/` — персонально, доступно во всех проектах;
- `<проект>/.claude/skills/spec-debate/` — в конкретном репозитории (можно коммитить).

Работает в обоих местах без правок: SKILL.md находит вспомогательный скрипт относительно
собственной папки скилла. Структура:

```
spec-debate/
├── SKILL.md
├── README.md
├── README_ru.md
├── LICENSE
├── scripts/
│   └── run_codex_critique.sh
└── evals/
    └── evals.json
```

## Использование

```
/spec-debate                       # цель — план/спека из текущего диалога
/spec-debate path/to/spec.md       # явный путь
/spec-debate path --xhigh          # максимум глубины рассуждений Codex (дефолт — high); заметно медленнее
```

Хочешь ещё раунд — просто вызови скилл снова: он подхватит состояние и пойдёт дальше.
Можно дописать свои инструкции в свободной форме — например «прогони 3 раунда» или
«гоняй, пока не останется значимых замечаний».

## Как это устроено (коротко)

1. Находит документ и называет его **тип и высоту** (ТЗ / дизайн / план) — это задаёт планку критики.
2. Запускает Codex как «беспощадного ревьюера» строго на этой высоте (вспомогательный скрипт
   `run_codex_critique.sh` передаёт промпт через stdin — документ не попадает в аргументы
   команды — и не даёт запуститься второму `codex exec`: одновременные запуски зависают).
3. **Ветит каждую находку**: проверяет, что она реальна (перечитывает раздел, грепает
   упомянутые файлы/числа), и оценивает пользу на высоте документа. Принимает целиком /
   частично / отклоняет — с однострочной причиной.
4. Вносит точечные правки, перечитывает изменённое на самосогласованность.
5. Пишет компактный отчёт, который легко просмотреть бегло (принято / частично / отклонено),
   и сохраняет состояние раунда.
