# spec-debate — дебаты двух моделей над планом решения задачи

Скилл для **Claude Code**. В каждом раунде **обе модели независимо предлагают улучшения** — Claude
и **вторая модель (OpenAI Codex)**, — а Claude выступает **редактором с правом вето**: проверяет каждое
предложение (и своё, и Codex'а), оставляет только то, что реально улучшает, остальное аргументированно
отклоняет.

Улучшается **всегда спека**, и зайти можно тремя способами:

- **задача** → Claude пишет спеку решения и дебатит её;
- **код для доработки** → Claude пишет *спеку модификаций*; сам код во время дебата не трогается —
  он применяется потом, отдельным шагом;
- **готовая спека / план / дизайн** → берётся как есть.

Можно и просто «посоветоваться с Codex» по узкому вопросу на ходу — скилл прогонит один
**prompt-only** проход и вернёт независимый разбор (проверенный Claude) без сохранения состояния.
Решишь продолжить — материализует файл спеки и перейдёт в стандартный режим дебатов.

> Критик, которого слушаются всегда, — это просто второй автор.
> Критик, с которым спорят, даёт результат лучше, чем каждая модель поодиночке.

По умолчанию один вызов = один раунд: предложения → вето → правки → отчёт → сохранённое
состояние (`.<имя>.debate-state.json` рядом с файлом). Нужно несколько раундов — вызови скилл ещё раз
(продолжит с того места, где остановились), либо попроси сразу: «прогони 3 раунда» или «гоняй, пока не
останется значимых замечаний». Зафиксированные находки передаются Codex'у, чтобы он не поднимал их
заново. Зашитый принцип — результат **лучше**, а не **больше**: скилл сопротивляется разрастанию
сложности.

## Требования

- **Claude Code** (CLI от Anthropic).
- **OpenAI Codex CLI** — установлен и авторизован:
  ```bash
  npm install -g @openai/codex
  codex login status    # проверка авторизации (если не залогинен — codex login)
  ```
  Нужен именно *независимый* второй движок — в этом весь смысл дебатов.
- **`bash` и `pgrep`** в PATH — их использует вспомогательный скрипт. Есть по умолчанию в **macOS и стандартных Linux**.
  **Windows: используйте WSL2** (обычное Linux-окружение) — это поддерживаемый путь. Нативный Windows не поддержан:
  под Git Bash `pgrep` обычно отсутствует, поэтому скрипт честно завершается с ошибкой (fail-closed), а без Git Bash
  bash вообще не запустит скрипт.

## Данные и приватность

По умолчанию spec-debate отправляет **полный текст спеки** (и файлы, которые Codex прочитает в рабочей папке) в OpenAI через Codex CLI. Для чувствительного материала включи **privacy-режим** («privacy mode» / «не отправляй код»): тогда Codex получает только одобренную абстрактную сводку — с пометкой, что доверие к выводам ограничено, — либо проход отменяется, если без материала вопрос не оценить. Не клади секреты в спеку и не запускай дефолтный режим на данных, которые нельзя передавать в OpenAI.

## Установка

Проще всего — если твоему Claude Code разрешены shell-команды — попросить его установить скилл:

> Установи https://github.com/OlegTestov/spec-debate в ~/.claude/skills/spec-debate

Он сам склонирует репозиторий в папку скиллов. Не сработало — склонируй вручную:

```bash
git clone https://github.com/OlegTestov/spec-debate ~/.claude/skills/spec-debate
```

Папку скилла можно положить в одно из двух мест:

- `~/.claude/skills/spec-debate/` — персонально, доступно во всех проектах;
- `<проект>/.claude/skills/spec-debate/` — в конкретном репозитории (можно коммитить).

Работает в обоих местах без правок: SKILL.md находит вспомогательный скрипт относительно
собственной папки скилла. Структура:

```
spec-debate/
├── SKILL.md
├── README.md
├── README_ru.md
├── LICENSE
├── scripts/
│   └── run_codex_critique.sh
└── evals/
    └── evals.json
```

## Использование

```
/spec-debate                       # цель — план/спека из текущего диалога
/spec-debate path/to/spec.md       # явный путь
/spec-debate path --xhigh          # максимум глубины рассуждений Codex (дефолт — high); заметно медленнее
```

Хочешь ещё раунд — просто вызови скилл снова: он подхватит состояние и пойдёт дальше.
Можно дописать свои инструкции в свободной форме — например «прогони 3 раунда» или
«гоняй, пока не останется значимых замечаний».

## Как это устроено (коротко)

1. **Выбирает режим и определяет спеку.** Быстрый осмотр объёма решает между одно-проходным
   *prompt-only* советом и *итерируемой спекой*; для задачи или кода Claude сначала пишет
   спеку / спеку модификаций по минимальной форме (goal, non-goals, ограничения, acceptance-критерии, …)
   и называет её **тип и высоту**.
2. **Обе модели предлагают независимо.** Claude формирует свой список улучшений; Codex предлагает
   свой, не видя списка Claude (чтобы не якориться). Вспомогательный скрипт `run_codex_critique.sh`
   передаёт промпт через stdin — документ не попадает в аргументы команды — и не даёт запуститься
   второму `codex exec` (одновременные запуски зависают). *(Опция `thorough`: +1 проход, где Codex ещё
   и оппонирует списку Claude.)*
3. **Сливает с правом вето.** Оба списка — Codex'а и собственный Claude — проходят одну процедуру:
   проверить, что находка реальна (перечитать раздел, грепнуть упомянутые файлы/числа), оценить пользу
   на высоте спеки, затем принять / частично / отклонить с однострочной причиной (отклонённые свои —
   тоже в отчёт).
4. **Вносит точечные правки**, затем перечитывает изменённое на самосогласованность.
5. **Пишет отчёт** (принято / частично / отклонено) и **сохраняет состояние**, рекомендуя СТОП, когда
   остаются лишь минорные находки. Для спеки модификаций применение к коду — отдельный пост-дебатный
   шаг, в отчёте отдельно от дебата.
